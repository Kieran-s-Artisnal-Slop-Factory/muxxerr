{{/* Administration: people, their running processes, their databases, and a log of what has been done to all three. Every action here is a separate one-button POST form. No checkboxes, no bulk actions, no menus. Admin screens in this shape are used a few times a year, usually to deal with something that has already gone wrong, and the failure mode worth designing against is "clicked the row above the one I meant" — not "took four clicks instead of two". The destructive ones carry data-confirm, which layout.html turns into a prompt when scripting is available and which the server must independently refuse to trust. Reads and writes are visually split: the account row shows state, the buttons at the end of it change state, and the export is a plain GET link because downloading somebody's database is a read and should behave like one (right-click, save as, resumable). The tables can be genuinely wide — a user with several apps, a running instance with pid and port — so each lives in its own overflow-x container. The page never scrolls sideways; the table does. The .wide wrapper is what widens the layout container, per layout.html. */}} {{define "title"}}Admin{{end}} {{define "content"}}
