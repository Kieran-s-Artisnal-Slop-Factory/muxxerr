# Adding an app

Everything muxxerr knows about an app is one block in
[apps.json](../../apps.json). This walks through adding a sync app, a static
app, and an app generated by `local-sync-template` (a separate generator repo,
not vendored here).

`apps.json` is parsed with `DisallowUnknownFields`, so a mistyped key is a
startup error that names the key. JSON has no comments — **do not add any, the
file must parse.** The annotated examples below are why this page exists.

## What makes an app multiplexable

Three requirements, all of which the existing apps happen to meet:

1. **Configured entirely by environment.** The supervisor sets `DB_PATH`,
   `PORT` and `STATIC_DIR` (and `TZ`, and anything in the app's `env` block).
   An app that reads a config file next to its binary cannot have per-tenant
   instances.
2. **Root-anchored routes.** The proxy strips `/<user>/<app>` before
   forwarding, so the child sees `/sync/pull`, exactly as if it were mounted at
   the origin root. An app that expects to know its own prefix will not work.
3. **A frontend that can be built with a base path.** For Astro, that is
   `--base`. The build must put a substitutable sentinel in every absolute URL;
   see the sentinel-rewrite section of
   [architecture.md](architecture.md).

An app that does background work has a fourth: it must be declared
`always_on`, because a lazily-started, idle-stopped process does nothing while
it is stopped.

## A sync app, annotated

This is readerr's entry with commentary. Strip the comments before pasting.

```jsonc
{
  // URL segment: /<user>/readerr/. Lowercase letters, digits and dashes, max
  // 31 chars. Cannot be a reserved name (login, admin, apps, sync, assets, …)
  // — see reservedNames in internal/config/config.go. Usernames are checked
  // against the same list, because the first path segment is a username.
  "name": "readerr",

  // Shown in the app chooser. Title defaults to name if omitted.
  "title": "Readerr",
  "description": "Reading list, weekly triage and notes.",

  // "sync" = Go backend + frontend. "static" = frontend only, no child process.
  "kind": "sync",

  // The app's repository root, relative to apps.json (absolute also works).
  "source": "../readerr",
  // Relative to source. Default "backend" and "frontend"; listed for clarity.
  "backend_dir": "backend",
  "frontend_dir": "frontend",

  // The sentinel the frontend was built with. muxbuild passes this to
  // `astro build --base=`, and the gateway substitutes it for the real
  // per-user prefix in HTML and JS responses. Empty disables rewriting
  // entirely — only correct if every asset URL is relative, which for an
  // Astro build it is not.
  "base_placeholder": "/__MUX__",

  // Polled after spawn until it answers; that is how "ready" is defined.
  "health_path": "/healthz",
  // Proxied by the admin export. Empty means "copy the database file", which
  // is riskier — see patches/05-hardening.md on torn backups.
  "backup_path": "/backup",
  // Filename inside data/instances/<user>/readerr/. Defaults to <name>.db.
  "db_file": "readerr.db",

  // THE FIELD PEOPLE GET WRONG. Every request path that is API rather than
  // frontend. These get no-store cache headers, are exempt from HTML/JS
  // rewriting, and are what the Referer shim recognises as a root-absolute
  // API call. Miss one and that endpoint 404s whenever the client calls it
  // root-absolutely. Trailing slash = prefix match; no slash = that path.
  // Cross-check against the app's route table in backend/main.go.
  "api_prefixes": ["/sync/", "/healthz", "/backup", "/title", "/dbsize"],

  // Extra environment for the child. DB_PATH, PORT and STATIC_DIR are always
  // set by the supervisor and WIN over anything here — a config typo cannot
  // point two users at one database.
  "env": {},

  // Stop the instance after this long with no requests. "0s" = never.
  "idle_timeout": "30m",
  // Start at gateway boot and never stop. Required for background work.
  "always_on": false,

  // SSRF policy for endpoints that fetch a caller-supplied URL. readerr's
  // title.go says outright that it has no SSRF hardening beyond a scheme
  // check — fine for a single-user LAN app, not fine once anyone with an
  // account can reach it. Policy is "block-private" (reject loopback,
  // RFC1918, link-local, unique-local) or "allow".
  "guarded_routes": [
    { "path": "/title", "param": "url", "policy": "block-private" }
  ]
}
```

### Finding `api_prefixes`

Read the app's route registrations. For readerr, `backend/main.go` lines 82-106:

```
GET  /healthz      POST /sync/push    GET /sync/pull
GET  /sync/stats   POST /sync/reset   GET /backup
GET  /title        GET  /dbsize
```

which collapses to `["/sync/", "/healthz", "/backup", "/title", "/dbsize"]`.
workoutt's (`main.go` lines 79-88) collapses to
`["/sync/", "/healthz", "/backup", "/push/"]`.

Anything the app registers under `/` after those is the static frontend
handler, and must **not** be listed.

### Choosing `idle_timeout` and `always_on`

Cold start is ~111 ms on a fresh database, ~55 ms warm, at ~12 MB RSS. Stopping
an idle instance is nearly free and restarting it is imperceptible, so the
default should be a short-ish timeout.

The exception is any app that does something while nobody is looking. workoutt
runs a 60-second ticker for push notifications, which fires only while its
process is alive — a stopped instance silently sends no reminders, and the user
has no way to tell. Hence:

```json
"idle_timeout": "0s",
"always_on": true
```

Both, together. `always_on` starts it at boot; `0s` stops the idle timer from
ever killing it. Setting one without the other is a bug you will find at 6 a.m.
when the reminder does not arrive.

Ask of any new app: *does anything happen in this process that no HTTP request
triggered?* Timers, schedulers, background reconciliation, outbound webhooks. If
yes, `always_on`.

## A static app

An app with no backend — a built site, an offline-only generated app, a set of
docs. No child process, no database, no port; the gateway serves `dist/` with
the sentinel rewritten, and that is all.

```json
{
  "name": "notes",
  "title": "Notes",
  "description": "Offline-only notes. No server, no sync.",
  "kind": "static",
  "source": "../notes",
  "frontend_dir": "frontend",
  "base_placeholder": "/__MUX__"
}
```

Everything else is omitted, and omitted is meaningful: `backend_dir`,
`health_path`, `db_file` and `api_prefixes` are only defaulted for
`kind: "sync"`, so a static app genuinely has none. `idle_timeout` and
`always_on` are meaningless with no process.

A static app still gets a per-user URL and still requires a login and an
instance row — the user "adds" it exactly like any other app. What it does not
get is per-user *data*: IndexedDB is per-origin, so two users on the same
browser profile share the offline-only app's storage. That is a real limitation
of static apps here, and the reason `kind: "static"` is not the easy way to
avoid running a backend.

## A generated (local-sync-template) app

Generated apps are structurally identical to workoutt and readerr — Astro
frontend, Go backend, `DB_PATH`/`PORT`/`STATIC_DIR` — so they mount the same
way. Their route table is the minimal one:

```json
{
  "name": "recipes",
  "title": "Recipes",
  "description": "Generated by local-sync-template.",
  "kind": "sync",
  "source": "../recipes",
  "backend_dir": "backend",
  "frontend_dir": "frontend",
  "base_placeholder": "/__MUX__",
  "health_path": "/healthz",
  "backup_path": "/backup",
  "db_file": "recipes.db",
  "api_prefixes": ["/sync/", "/healthz", "/backup"],
  "env": {},
  "idle_timeout": "30m",
  "always_on": false,
  "guarded_routes": []
}
```

Three notes specific to generated apps:

- **`api_prefixes` is exactly those three.** The generator emits no `/title`,
  `/dbsize` or `/push/` — those are readerr and workoutt extensions. If you have
  hand-added an endpoint since generating, add it here too.
- **`guarded_routes` is empty and should stay empty** unless you added an
  endpoint that fetches a caller-supplied URL. The generated backend has none.
- **`always_on: false` is correct.** The generated backend does no background
  work: no tickers, no schedulers.

Generated apps are multiplex-ready out of the box: the current
`local-sync-template` already excludes the API from caching, isolates its cache
keys by prefix, and derives `getSyncUrl()` from `BASE_URL` — the fixes that used
to be proposed here as patches. The only outstanding generator change is a
convenience (have it emit this `apps.json` block for you), tracked in
[patches/04-generator.md](../../patches/04-generator.md) §4.4.

For an **offline-only** generated app (`includeBackend: false`), use the static
form above instead: it has no `backend/` directory to compile.

## Build and run

```bash
go run ./cmd/muxbuild -config apps.json -only recipes   # just this one
go run ./cmd/mux -config apps.json
```

`muxbuild` compiles `<source>/<backend_dir>` to
`runtime/apps/<name>/<name>[.exe]` and runs
`astro build --base=<base_placeholder>` in `<source>/<frontend_dir>`, copying
the result to `runtime/apps/<name>/dist`. Rebuild after changing an app's
source **or** its `base_placeholder`; the sentinel is baked into the build.

`muxbuild` also records the app's commit (from its git checkout) and version
(from an optional `VERSION` file in the app's source root) to
`runtime/apps/<name>/build.json`, which the dashboard shows as a small badge on
the app's card; and if the source root has a `CHANGELOG.md`, it renders that to
HTML for a **Release notes** modal on the card. Nothing to configure — no
`VERSION`/`CHANGELOG.md` and no git just means no badge and no button. See
[versioning.md](versioning.md).

Then add the app to your account from the chooser at `/`. Adding it creates
`data/instances/<you>/recipes/` and the instance row; nothing exists until you
do.

## Checklist for a new app

- [ ] Name is lowercase, URL-safe, and not reserved (`config.Reserved`)
- [ ] `source` resolves relative to `apps.json`
- [ ] `api_prefixes` matches the backend's route table exactly
- [ ] `always_on` is true if and only if the app does background work
- [ ] `guarded_routes` covers every endpoint that fetches a caller-supplied URL
- [ ] The app reads its database path from `DB_PATH` and binds `PORT`
- [ ] Nothing in `env` collides with `DB_PATH`, `PORT` or `STATIC_DIR`
- [ ] `muxbuild` succeeds and `runtime/apps/<name>/dist/index.html` contains the
      sentinel string

## When it does not work

| Symptom | Likely cause |
|---|---|
| Blank page, 404s for `/_astro/*` | frontend built without `--base` (used `npm run build`, not `muxbuild`), or `base_placeholder` does not match what it was built with |
| App loads, sync 404s | endpoint missing from `api_prefixes` |
| Sync works in one browser, not another | the `Referer` shim losing the header — only affects an *unpatched* app; the vendored apps derive the sync base from `BASE_URL` and don't rely on the shim (see [patches/README.md](../../patches/README.md)) |
| Instance never becomes ready | `health_path` wrong, or the child is failing at startup — check the gateway log for the child's stderr |
| Background work stops overnight | `always_on` missing, or set without `idle_timeout: "0s"` |
| Startup error naming a field | `DisallowUnknownFields` — a typo, or a `kind: "static"` app given a sync-only key |
