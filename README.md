# muxxerr

An authentication gateway that turns single-tenant, no-auth, local-first apps
into multi-tenant ones — without modifying them.

[workoutt](apps/workoutt) and [readerr](apps/readerr) — vendored here as git
submodules — and anything generated by local-sync-template are all built the
same way: a static Astro frontend that owns its data in IndexedDB, plus a small
Go server that is a sync target, configured entirely by `DB_PATH`, `PORT` and
`STATIC_DIR`, with no users, no login and no concept of a second person. That
is the right design for one person on one laptop, and a dead end the moment you
want to hand a URL to a friend.

This gateway takes that design at face value instead of arguing with it. Rather
than adding auth *inside* each app, it runs **one isolated child process and one
SQLite file per (user, app) pair** and puts a login in front of them.

## Architecture in five sentences

`mux` is a reverse proxy that owns `/`, authenticates a session cookie, and
mounts each user's instance at `/<username>/<app>/`. A supervisor starts an
app's backend on demand — cold start is about 111 ms on a fresh database, 55 ms
warm, at roughly 12 MB RSS — with `DB_PATH` pointing at
`data/instances/<user>/<app>/<app>.db`, a private port, and per-child `TZ`, then
proxies to it and stops it again after its idle timeout. Because the apps'
routes are root-anchored (`/sync/push`, `/healthz`, `/backup`), the proxy strips
the `/<username>/<app>` prefix on the way in. Because their frontends are built
once with `astro build --base=/__MUX__`, every internal URL in the built output
contains that sentinel, which the gateway rewrites to the real per-user prefix
on the way out — one build serves every user. Identity lives in a single
`data/mux.db` (Argon2id + a pepper stored outside the database), and no app ever
learns that any of this is happening.

Longer version, with the reasoning and the rejected alternatives:
[docs/dev/architecture.md](docs/dev/architecture.md).

## Quick start

### With Docker (nothing to install but Docker)

```bash
git clone --recurse-submodules https://github.com/<you>/muxxerr
cd muxxerr
docker compose up -d
```

That is the whole thing. The image builds the apps and serves them; `./data`
on the host holds every database. Change the port with `MUX_PORT` in a `.env`.
HTTPS, backups and upgrades: [docs/admin/deployment.md](docs/admin/deployment.md).

### From source

Requirements: Go 1.25+, Node 20+ (for the app frontends), and git.

```bash
git clone --recurse-submodules https://github.com/<you>/muxxerr
cd muxxerr

# 1. Build every configured app: compile each Go backend, and build each
#    frontend with --base=/__MUX__ into runtime/apps/<name>/dist.
go run ./cmd/muxbuild

# 2. Run the gateway.
go run ./cmd/mux
```

Already cloned without `--recurse-submodules`? `git submodule update --init`.

Open <http://localhost:8080>, sign up, and **write down the recovery passphrase
it shows you once** — there is no email server here, so that passphrase is the
entire account-recovery story. The first account created is an admin.

Neither command needs `-config`: both look for `apps.json` in the working
directory, then beside the binary. `-config` is for when it lives elsewhere.
`muxbuild -only <name>` rebuilds one app, which is what you want while
iterating. `PORT` overrides the configured listen port, for containers.

## The URL scheme

| URL | What it is |
|---|---|
| `/` | app chooser (logged in) or the login page |
| `/login`, `/logout`, `/signup`, `/reset` | gateway auth |
| `/account` | change your password, view sessions |
| `/apps/<app>/logs` | the last 50 lines your instance wrote |
| `/tools/sqlite` | browse a snapshot of a database, in your browser |
| `/tools/sql` | run statements against a live database (opt-in) |
| `/apps/<app>/export` | a consistent snapshot of your own database |
| `/admin` | user management, exports, audit log (admins only) |
| `/<username>/<app>/` | one user's instance of one app |
| `/<username>/<app>/sync/pull` | that instance's backend, proxied |
| `/<username>/<app>/manifest.webmanifest` | served **without** a session — see below |

The first path segment is a username, which is why the gateway reserves a list
of names (`login`, `admin`, `assets`, `apps`, …) that can be neither a username
nor an app name — see `reservedNames` in
[internal/config/config.go](internal/config/config.go).

Apps are opt-in per user: an instance directory and database are created the
first time you add an app from the chooser, not at sign-up.

One narrow exception to "everything under `/<username>/<app>/` needs a session":
the PWA manifest and the app's icons. A browser fetches `<link rel="manifest">`
with credentials *omitted*, so requiring a cookie there 401s the fetch and makes
every app permanently un-installable. Those files are build artifacts —
byte-identical for every user, containing only the app's name and colours — so
the gateway serves them straight off disk, before the auth check and without
starting a child process. The list is exact and the app shell and sync API are
not on it: [internal/gateway/public.go](internal/gateway/public.go).

## Configuration

One file, [apps.json](apps.json), read by both commands. It declares the listen
address, where data and build output live, and one block per app: where its
source is, which paths are API rather than frontend, how long it may idle, and
whether it must stay running.

An app's `source` can be a **submodule** (the default — `apps/readerr`, pinned
to an exact commit by this repo), a **local path** (`../readerr`, for when you
are editing it), or a **git URL** fetched at build time
(`git+https://github.com/you/readerr`, optionally `#tag`). They differ in who
decides which version gets built:
[docs/dev/app-sources.md](docs/dev/app-sources.md).

Adding an app — including a static, backendless one — is
[docs/dev/adding-an-app.md](docs/dev/adding-an-app.md).

## Compatibility with unmodified apps

An app whose client calls its API root-absolutely (`fetch('/sync/pull')`,
because `getSyncUrl()` falls back to `''`) has no user or app in the request. The
gateway handles that with a `Referer`-based shim: a root-absolute request to a
known API path, sent from a page under `/alice/readerr/`, is re-attributed to
that instance. It works, and it is why you can point this at an *unmodified*
checkout.

The vendored apps are no longer unmodified: the sync-URL and service-worker fixes
that used to live in [patches/](patches/) (01–03) are **applied upstream now**,
so readerr and workoutt send the prefixed URL directly and the shim is only a
fallback for forks that lack the fix. What remains in `patches/` is one generator
convenience (04 §4.4) and the backend hardening set (05) — see
[patches/README.md](patches/README.md).

## What this is not

**Not a security boundary against the apps themselves.** Every child process
runs as the same OS user, with the same filesystem access, as the gateway.
Isolation is one process and one database per tenant — good enough to stop
users seeing each other's data through the app, useless against an app that is
actively malicious. Do not mount code you would not run yourself.

**Not multi-user *within* an app.** There is no sharing, no collaboration, no
"send this to Bob". Each user gets a private single-tenant app that happens to
live at a URL with their name in it. Two users of readerr have nothing in
common but the binary.

**Not a scalable hosting platform.** One process per active (user, app) pair,
each holding an open SQLite file, is fine for a household or a small team and
obviously wrong for a thousand users. There is no clustering, no shared
storage, no horizontal anything. Idle timeouts are the only capacity management,
and `always_on` apps (workoutt, whose notification scheduler only fires while
the process is alive) opt out of even that.

**Not hardened for the open internet.** It ships plain HTTP with
`secure_cookies: false` by default because the expected deployment is a LAN or
a Tailscale network. Put it behind TLS and flip that flag before exposing it.
readerr's `/title` endpoint is SSRF-guarded by the gateway, but the apps behind
it were written with "no auth, LAN posture" in their header comments and that has
not stopped being true — and the child backends currently bind *all* interfaces,
not just loopback, so a from-source deployment leaves each unauthenticated app
reachable on the LAN. Two shipped defaults also want a second look before you
expose this: the live SQL console (`sql_console: true`) and open sign-ups. The
full list, with severities and fixes, is in
[docs/improvements/](docs/improvements/) — **read it before putting this on a
network you do not control.**

**Not a backup system.** `/backup` gives you a database file; nothing schedules
it, rotates it, or takes it off the machine. See
[docs/admin/operations.md](docs/admin/operations.md), which also explains why
losing `data/pepper.key` means every account on the system is unrecoverable.

**Not true isolation inside a shared browser.** This is the one limitation of
the `/<user>/<app>/` URL scheme worth understanding before you deploy it.
IndexedDB, localStorage and Cache Storage are scoped to an *origin*, never to a
path — so `/kieran/readerr/` and `/alex/readerr/` are the same storage as far as
the browser is concerned. On the server the two are entirely separate databases;
in one browser profile they are not.

The gateway injects a small script into every app page that notices when the
signed-in user has changed and clears local state before the app boots, after
which the app re-syncs from its own backend. That makes sequential logins on a
shared device safe, which is the case that actually happens. It does not make
two people using one browser profile at the same time safe, and nothing inside
this URL scheme can — that needs an origin per user, i.e. subdomains. See
[docs/dev/architecture.md](docs/dev/architecture.md).

## Database tools

**Tools → SQLite viewer** loads a snapshot of one of your app databases — or a
`.db` file you drag in — into SQLite compiled to WebAssembly, and runs your
queries *in the browser*. Nothing typed there reaches the server, so there is no
list of forbidden keywords to get wrong: the isolation is structural. Export the
result back out as `.db` or as SQL.

**Tools → SQL console** is the opposite, and is off unless `apps.json` sets
`"sql_console": true`. It runs statements against the real file an app reads,
with no undo and no automatic backup. Every visit makes you type the warning
sentence, and the instance is stopped before anything runs so there is only one
writer.

They are separate pages on purpose. Put a "write mode" checkbox on one page and
the safe operation starts carrying the dangerous one's warnings — which is how
warnings stop being read. [docs/dev/database-tools.md](docs/dev/database-tools.md).

## The gateway's own UI

Server-rendered Go templates, one hand-written stylesheet, no build step and no
external requests. The mark — one node branching into several — is what the
thing does: a single entry point fanning out to isolated per-user instances.
It is `internal/web/static/icon.svg`, with a deliberately simpler `favicon.svg`
for 16px and raster fallbacks regenerated by `go run ./cmd/muxicons` when the mark
changes.

Static assets are content-addressed (`/_mux/static/<hash>/mux.css`) and served
`immutable`. They are baked into the binary, so "has this changed" and "how long
may the browser keep it" have the same answer — and without the hash, upgrading
the gateway leaves everyone on the previous stylesheet until their cache expires.
